# XEMK 项目梳理、问题排查与面试问答

## 1. 项目定位与整体目标

这个项目是一个双人对战卡牌游戏原型，核心由三部分组成：

1. C++ WebSocket 游戏服务端（规则、回合、结算、状态同步）
2. 前端客户端（Web 版 + PyQt 桌面版）
3. 第三方依赖（`websocketpp`、`asio`、`nlohmann/json`）

从代码实现看，它已经具备“可联机对战”的主流程，但规则系统、内存与状态管理、协议一致性还处在持续迭代阶段。

---

## 2. 目录与主要文件说明

项目根目录主要关注这些文件：

1. `main1.cpp`  
作用：程序入口，创建 `GameServer` 实例并阻塞等待输入退出。

2. `server1_8_1.hpp`  
作用：服务端核心。负责 WebSocket 连接管理、消息分发、回合推进、发牌、对战结果广播。

3. `play3_6_2.hpp`  
作用：战斗结算与出牌解析。包含槽位解析、对攻逻辑、卡牌死亡处理。

4. `card3_6_2.hpp`  
作用：卡牌模型与卡池。定义 `Card`、`Cardfactory`、`CardRandomizer`。

5. `game_interface1_8.html`  
作用：Web 客户端，负责展示手牌、拖拽上场、发送 WebSocket 消息。

6. `game_interface.py`  
作用：较完整 PyQt 客户端版本（UI 更复杂、日志更丰富）。

7. `game_interface_desk.py`  
作用：另一版 PyQt 客户端（结构与 `game_interface.py` 高度重合）。

8. `third_party/`  
作用：第三方库源码（包含 `asio`、`asio_1_18`、`websocketpp`、`nlohmann/json.hpp`）。

## 3. 运行主流程（端到端）

### 3.1 启动

1. `main1.cpp` 创建 `GameServer`。  
2. `GameServer` 在 `8002` 端口监听并启动后台线程运行 `ws_server_.run()`。

### 3.2 玩家接入

1. 客户端发送 `player_join`。  
2. 服务端在 `handle_player_join` 里登记 `player_id -> connection_hdl`。  
3. 两人都加入后，调用 `generate_unique_numbers()` 发初始手牌，再广播 `game_start`。

### 3.3 出牌与实时放置同步

1. 客户端拖拽卡牌时会发送 `card_placement_update`（`add/remove/clear`）。  
2. 服务端在 `on_message` 的 `card_placement_update` 分支处理（当前只显式处理 `add/clear`）。  
3. 服务端会同步 `player_bonus`（骨头等资源）与己方/对方场面信息。

### 3.4 回合提交与战斗结算

1. 客户端点击“结束己方回合”，发送 `player_action`，附 `slots`。  
2. 服务端调用 `handle_player_action`，再进入 `game_play.an_slot_card(...)` 解析上场卡牌。  
3. 第二位玩家结束回合后，调用 `game_play.cur_plays(...)` 对攻结算。  
4. 服务端广播 `player_hp`，若满足结束条件再广播 `game_end`。

### 3.5 新回合与重开

1. 客户端发送 `start_new_round`。  
2. 双方都同意后 `reset_game()` + `generate_unique_numbers()` 并再次开局。

## 4. 核心代码功能拆解

## 4.1 卡牌模型与卡池（`card3_6_2.hpp`）

### `Card`

主要字段：

1. 基础属性：`name/HP/ATK/property/cost/race`
2. 运行时状态：`card_id`（唯一编号）、`card_state`（是否已上场）
3. 临时状态：`hp_temp_save/atk_temp_save`（为属性效果做回滚）

关键方法：

1. `toJson()`：序列化卡牌到前端协议结构（含 cost 数组）
2. `set_play_current_card_id/get_play_current_card_id()`：发牌后分配唯一 id
3. `lossHP/setHP/setATK/getATK()`：生命与攻击调整

### 工厂体系 `Cardfactory + 各种族工厂`

采用简单工厂模式封装卡牌创建流程，`createCardwithsetup()` 先创建再 `setupCard()`。

### `CardRandomizer`

职责：

1. 初始化固定卡池（牛蛙、郊狼、黑山羊等）
2. 提供 `getRandomCard()`、`getcard(name)`、`getsquirrel()` 返回可上手牌
3. 分配全局递增 `card_id`，用于前后端唯一识别

---

## 4.2 战斗与出牌解析（`play3_6_2.hpp`）

### `an_slot_card(...)`

根据前端传来的 `slots` 数据，在 `player_cards_` 中按 `card_id` 找出对应卡牌，做费用检查后放入槽位。

当前实现要点：

1. 主要处理“骨头消耗”逻辑
2. 用 `card_state` 防止已在场上的牌重复扣费
3. 将解析结果写回 `slots_cards[slot_index]`

### `cur_plays(...)`

四个槽位逐一结算，核心顺序：

1. 先手位攻击后手位
2. 处理后手死亡
3. 再由后手反击先手
4. 处理先手死亡
5. 角色血量 `character_HP` 累加/递减，判断胜负阈值

### `delate_cards(...)`

卡牌死亡时，从手牌容器移除，并从场上容器删除并 `delete` 指针，同时给骨头资源 +1。

### `process_attribute_atk(...)`

设计上用于处理“死神之触/空袭/急袭/脆骨/全向打击”等特性；当前在主结算里调用被注释，意味着属性系统未完整接入实际战斗。

---

## 4.3 服务端与协议（`server1_8_1.hpp`）

### `GameServer` 主要职责

1. WebSocket 生命周期管理：`on_open/on_close/on_message`
2. 玩家注册与重连管理：`player_connections_`、`disconnected_players_`
3. 回合与资源状态维护：`flag/choosing_card/xianjiing/last_player_bones/...`
4. 向前端同步：`numbers_assigned/move_accepted/opponent_move/player_hp/player_bonus/game_end`

### 关键消息类型

1. `player_join`
2. `card_placement_update`
3. `player_action`
4. `special_action`
5. `start_new_round`

### 同步函数

1. `send_cards_to_player`：发手牌
2. `process_player_move`：发己方场面
3. `notify_opponent_move`：发对方场面
4. `broadcast_game_start`：开局广播

---

## 4.4 客户端（HTML + PyQt）

共同能力：

1. WebSocket 连接与重连
2. 手牌渲染、拖拽放槽位
3. 发送 `card_placement_update` 与 `player_action`
4. 接收并渲染 `numbers_assigned/opponent_move/player_hp/...`

差异：

1. `game_interface1_8.html`：浏览器版本，便于快速联调
2. `game_interface.py`：桌面端 UI 更完整
3. `game_interface_desk.py`：另一版桌面端，存在与 `game_interface.py` 重复实现

## 5. 已识别问题清单（按严重度）

以下问题是基于当前代码直接观察得到，含定位点（文件与行号）。

## P0（高风险，可能崩溃或严重错乱）

1. **存在明显内存泄漏风险（原始指针生命周期不完整）**  
位置：`card3_6_2.hpp:372`, `card3_6_2.hpp:386`, `card3_6_2.hpp:402`, `server1_8_1.hpp:639-666`  
说明：`CardRandomizer` 多处 `new Card` 返回裸指针，`generate_unique_numbers()` 清空手牌容器时未统一释放旧指针；`reset_game()` 未清理旧槽位中的动态对象。长期运行可能内存持续增长。

2. **协议分支不一致：前端发送 `remove`，服务端只处理 `add/clear`**  
位置：服务端 `server1_8_1.hpp:198`, `server1_8_1.hpp:235`；前端 `game_interface.py:960`, `game_interface_desk.py:841`, `game_interface1_8.html:1710`  
说明：会导致部分“单卡移除”行为在服务端被忽略，形成前后端状态不一致。

3. **属性系统主流程未启用**  
位置：`play3_6_2.hpp:186`, `play3_6_2.hpp:215`（调用注释）  
说明：虽然定义了大量属性，但实际战斗结算未执行属性逻辑，游戏规则与设计不一致。

## P1（中高风险，逻辑正确性问题）

1. **状态字段混用全局成员，耦合度高，易串状态**  
位置：`server1_8_1.hpp:70-76`, `server1_8_1.hpp:174-177` 等  
说明：`player_idnex/player_idnex_op/flag/choosing_card/xianjiing` 等为 `GameServer` 共享字段，消息处理中多处复写，复杂场景下易出现跨玩家或跨消息污染。

2. **`an_slot_card` 费用解析仅看 `cost_map.begin()`**  
位置：`play3_6_2.hpp:277-296`  
说明：若卡牌费用有多项（`multimap`），只处理第一项，规则不完整。

3. **胜负阈值边界不对称且可疑**  
位置：`play3_6_2.hpp:195`, `play3_6_2.hpp:231`  
说明：使用 `> 5` 与 `< -5`，不是 `>= 5` / `<= -5`；可能导致“刚好 5/-5 不结束”的边界 bug。

4. **`reset_game()` 使用当前玩家字段发消息，有时机依赖**  
位置：`server1_8_1.hpp:867-899`  
说明：这里使用 `player_idnex/player_idnex_op` 发 `player_hp/player_bonus`，若字段未按预期更新，可能发送目标错误。

5. **等待重开提示发给了对手而不是请求方**  
位置：`server1_8_1.hpp:632` 附近逻辑  
说明：注释里也有 `send_to_player(player_id, ...)` 的痕迹，当前实现是发给 `opponent_id`，易导致交互体验混乱。

## P2（可维护性与工程化问题）

1. **服务端头文件过重、职责过多**  
`server1_8_1.hpp` 同时管理网络、会话、规则编排、同步协议，后续扩展成本高。

2. **命名与拼写不一致**  
例如 `player_idnex`、`delate_cards` 等，影响可读性。

3. **调试注释和历史问题记录大量堆积**  
虽有帮助，但长期会干扰主逻辑阅读，建议迁移到 issue 或 changelog。

4. **客户端重复实现较多**  
`game_interface.py` 与 `game_interface_desk.py` 大量重复，可抽公共模块。

5. **缺少自动化测试与构建脚本标准化入口**  
项目根目录未见统一的 CMake/测试入口，回归依赖手工。

## 6. 面试可能会问到的技术问题与详细答案

## 6.1 架构与通信

1. **问：这个项目为什么用 WebSocket，而不是 HTTP 轮询？**  
答：这是回合制但包含实时拖拽同步和对手状态更新。WebSocket 可保持长连接，双向推送延迟低、消息模型自然，避免轮询带来的高延迟与冗余请求。

2. **问：你们如何定义前后端协议？**  
答：以 `type` 字段区分事件（如 `player_join`、`player_action`、`opponent_move`）。负载统一 JSON。服务端在 `on_message` 分派，前端在 `handle_message` 分派。后续建议加入协议版本号和 schema 校验。

3. **问：如何支持断线重连？**  
答：服务端用 `player_connections_` 保存会话、`disconnected_players_` 标记断开玩家；重连时替换连接句柄并重发当前状态，再通知对手 `opponent_reconnected`。

4. **问：消息乱序或重复怎么处理？**  
答：当前版本主要依赖回合状态机和 `card_id` 唯一标识，尚未实现“消息序号 + 幂等去重”。生产化应增加 `message_id/turn_id` 并在服务端做幂等检查。

## 6.2 游戏规则与状态机

5. **问：回合状态是怎么推进的？**  
答：核心由 `flag + choosing_card + last_player` 协同推进：两名玩家依次提交 `player_action`，第二名提交后触发战斗结算，再进入特殊选牌阶段。后续建议重构为显式枚举状态机（如 `WaitingJoin/Player1Turn/Player2Turn/Resolve/ChooseCard`）。

6. **问：为什么卡牌需要 `card_id`？**  
答：前端拖拽/清除动作必须定位到同一张实例卡。名称不唯一，内存地址也不能跨进程传递，因此服务端给每张实例分配 `card_id`，前端以此回传操作对象。

7. **问：你们如何处理卡牌死亡？**  
答：在 `cur_plays` 中当血量 <= 0 调 `delate_cards`，从手牌容器移除、从槽位删除并释放对象，同时增加骨头资源。

8. **问：属性系统如何接入？**  
答：已封装在 `process_attribute_atk`，包含“急袭/空袭/全向打击/死神之触”等，但主流程调用被注释。下一步应在攻防前后插入属性钩子，并补充规则测试。

## 6.3 C++ 设计与内存管理

9. **问：这个项目用了哪些 C++ 设计模式？**  
答：卡牌创建使用了工厂模式（`Cardfactory` + 各种族工厂）；网络消息分发使用事件回调模式；卡牌状态管理接近状态模式但还未完全抽象。

10. **问：为什么当前内存风险较高？**  
答：核心原因是裸指针跨多个容器共享，生命周期分散在不同函数中，容易漏删或重复删除。更稳妥做法是 `unique_ptr` 独占所有权 + 句柄索引访问，或对象池集中管理。

11. **问：如何重构内存模型？**  
答：建议：
   1. 卡牌统一由 `CardStore` 持有 `std::unordered_map<CardId, std::unique_ptr<Card>>`
   2. 手/场容器只存 `CardId`
   3. 删除只在 `CardStore` 做一次
   4. 协议全程传 `CardId`

12. **问：为什么不建议在容器里直接 `new/delete`？**  
答：因为很难保证异常安全和所有路径释放，尤其在多阶段逻辑（发牌、结算、重置、断线）下。RAII + 智能指针可显著降低泄漏和悬挂指针风险。

## 6.4 并发与线程

13. **问：这个服务端是多线程吗？线程安全吗？**  
答：WebSocket 事件主循环在一个后台线程，主线程只做阻塞等待退出。代码中有 `connections_mutex_` 与 `game_mutex_` 做基本保护，但状态变量仍较分散，建议统一加锁边界并减少共享可变状态。

14. **问：为什么要避免“锁内发送网络消息”？**  
答：锁内做 IO 会拉长临界区，增加阻塞和死锁风险。应先构造快照数据，释放锁后再发送。

## 6.5 前端与工程实践

15. **问：前端为什么有三个版本？**  
答：Web 版便于快速联调，PyQt 版更接近桌面发行。当前是原型迭代阶段导致双实现并存，后续应收敛到一个主客户端，另一个作为调试工具。

16. **问：你如何保证前后端协议一致？**  
答：当前靠手工约定，已暴露 `remove` 不一致问题。工程化建议引入共享 schema（JSON Schema/Protobuf），并做契约测试。

17. **问：`asyncio.run_coroutine_threadsafe` 使用要注意什么？**  
答：必须提交到正确的事件循环。当前代码多处直接 `asyncio.get_event_loop()`，在跨线程上下文可能拿到非预期 loop，建议显式保存 websocket 线程 loop 并统一提交。

## 6.6 可测试性与上线能力

18. **问：这个项目怎么做测试？**  
答：建议分层：
   1. 规则层单元测试（结算、费用、属性）
   2. 协议层契约测试（消息字段与状态迁移）
   3. 双客户端端到端回放测试（固定随机种子）

19. **问：如果线上出现“状态不同步”，你怎么排查？**  
答：加三类日志：
   1. 入站消息日志（含 player_id、turn_id、card_id）
   2. 状态快照日志（手牌/槽位/血量/资源）
   3. 出站消息日志（发给谁、内容摘要）  
通过同一 `turn_id` 串联客户端与服务端日志定位分叉点。

20. **问：如何让这个项目更接近可发布版本？**  
答：优先级：
   1. 重构内存与状态机
   2. 修复协议一致性
   3. 抽离规则引擎并补测试
   4. 引入构建脚本与 CI
   5. 收敛客户端实现
